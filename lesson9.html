Paso 1: Romper el diseño gráfico en una jerarquía de componentes

  • ¿Cómo decidimos que elementos del diseño son componentes?
  • de la misma manera que deciden como crear clases y funciones
  • de la misma manera que un diseñador organiza su trabajo
  • de la misma manera que un desarrollador diseña el API
  1. FilterableProductTable (orange): contains the entirety of the example
  2. SearchBar (blue): receives all user input
  3. ProductTable (green): displays and filters the data collection based on user input
  4. ProductCategoryRow (turquoise): displays a heading for each category
  5. ProductRow (red): displays a row for each product
  • FilterableProductTable
    • SearchBar
    • ProductTable
      • ProductCategoryRow
      • ProductRow

Paso 2: Diseño estático de la aplicación

  • Se crean clases de cada uno de los componentes
  • En una versión estática de un sitio, toda la información se pasa por props
  • No utilizen estado, el estado es para interactividad y dinamismo
  • top-down o bottom-up
  • Los componentes sólo can a tener metodos render

Paso 4: ¿Cuál componente debe ser el contenedor de este estado?

  • ¿Qué componentes renderizan algo en base a este estado?
  • Se debe encontrar un componente que contenga a todos estos
  • Si no hay un componente contenedor, se puede crear uno
  • ProductTable needs to filter the product list based on state and SearchBar needs to display the search text and checked state.
  • The common owner component is FilterableProductTable.
  • It conceptually makes sense for the filter text and checked value to live in FilterableProductTable
